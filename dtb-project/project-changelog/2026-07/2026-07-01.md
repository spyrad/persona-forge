### Implementiert

Zwei zusammenhängende Prod-Themen um `ENCRYPTION_KEY` gelöst: erst das fehlende
Worker-Secret (F3-Follow-up), dann ein dadurch ausgelöster Krypto-Key-Mismatch, der alle
Läufe scheitern ließ. Systematische Diagnose (Symptom → Code-Pfad → Wurzel), kein Rätselraten.

- **F3-Follow-up — `ENCRYPTION_KEY`-Sync:** Prod zeigte Banner „ENCRYPTION*KEY is not
  configured — model configurations cannot be saved" (`config-status.ts:17`). Ursache: der
  Deploy-Job in `.github/workflows/ci.yml` synct nur `SUPABASE*\*`/`SENTRY_DSN`in den Worker,
**nicht**`ENCRYPTION_KEY`. Fix: `ENCRYPTION_KEY`in`secrets:`-Liste **und** `env:`-Block
des `wrangler-action`-Steps aufgenommen (`93fae0c`). GitHub-Secret gesetzt, gepusht → Deploy
  grün, **Banner-Vorkommen 0** im Prod-HTML verifiziert (Prod 200).
- **Incident — Läufe scheitern „Something went wrong" / hängen „Läuft 0/5, 0 Tokens":**
  Diagnose ergab **AES-GCM-Key-Mismatch**. `getDecryptedTarget` (`model-configs.ts:122-138`)
  ruft `decryptApiKey` **außerhalb** des try/catch von `processNextRepetition`
  (`runs.ts:348`); wirft der Decrypt, fliegt der Fehler → Step 500, **kein Rep geschrieben** →
  Lauf hängt bei 0 (provider-unabhängig, weil vor dem LLM-Call).
- **Wurzel:** `SUPABASE_URL` = **gehostete** Supabase-DB (`lccaundrniuievkmusko.supabase.co`),
  von Prod **und** lokalem Dev geteilt. Die Modellkonfigs wurden mit dem **`.dev.vars`-Key**
  verschlüsselt; der von mir vorher gesetzte **Zufalls-Key** konnte sie nicht entschlüsseln.
  Mein „frischer Key ist gefahrlos" war falsch — Ciphertext existierte längst in der geteilten DB.
- **Fix:** GitHub-Secret `ENCRYPTION_KEY` auf den **`.dev.vars`-Wert** überschrieben (User),
  Re-Deploy via Push `70784f7` → grün. `.env` == `.dev.vars` verifiziert (beide 44 Zeichen,
  identisch). Zielzustand: `.dev.vars` = `.env` = GitHub-Secret identisch.
